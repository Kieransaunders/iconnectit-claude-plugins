# Phase 0 — Prep & Emitter Hardening

> Folded into GSD from `skills/landing-page/RENDER-FAULT-FINDER-SPEC.md` §6 (Phase 0)
> plus the agreed emitter hardening. All paths below are relative to the divi5-tools repo
> root; the code lives under `skills/landing-page/scripts/`.

## Goal

Make the two cheap structural changes the spec's Phase 0 calls for **and** close the
faults at source in the generator, so Tier 1 (`render-safety.js`, Phase 1) lands as a
tiny diff over a code base that can no longer *emit* faults 1, 3 and 8 in the first place.

Two intents, kept distinct:

1. **Enablement (spec §6 Phase 0).** Refactor `validate.js` so a future
   `render-safety.js` can re-walk parsed tokens without re-parsing, and so the
   hard-coded dash ban becomes a configurable glyph set that module can own. Zero
   change to current validator behaviour.
2. **Emitter hardening (agreed addition).** Guarantee `divi-builder.js` cannot produce
   the deterministic render faults: `theatreAttrs()` becomes the *only* writer of the
   custom-attributes path, and all module HTML content goes through one escaping helper
   whose output is proven to survive Divi's import round-trip.

## Why this scope (trade-off note)

`block()` already runs every attrs object — including `innerContent` HTML — through
`JSON.stringify` (`divi-builder.js:63`), and `theatreAttrs()` already writes the
canonical `module.decoration.attributes.desktop.value.attributes` array
(`divi-builder.js:111`). So the generator is *mostly* already correct. The residual
risk is the two `attrs:` escape hatches (any caller can hand-merge a wrong-shape
attribute branch) and the unconfirmed exact mechanism of the raw-quote fault. Phase 0
removes the escape-hatch risk and **confirms** the raw-quote mechanism before any
auto-fix is designed in Phase 1 — rather than shipping a fixer against a guessed cause.

<read_first>
- skills/landing-page/scripts/validate.js  (content loop ~95-194; `parseBlocks`, `contents`, the per-token walk)
- skills/landing-page/RENDER-FAULT-FINDER-SPEC.md §4 "Required tweak to validate.js" (lines 92-98), "Input model" (100-115)
</read_first>

<action>
In the existing `for (const { key, content } of contents)` loop, capture the result of
`parseBlocks(content)` into a module-scope map `tokensByKey` keyed by the content `key`
(`tokensByKey[key] = tokens`). Do not change the inline walk or any existing
`err`/`warn`/`pass` call. After the loop, `tokensByKey`, `contents`, and `doc` must be
referenceable as a single bundle (e.g. `{ doc, contents, tokensByKey }`) so a future
`require('./render-safety')(...)` call can consume them — but do **not** add the
render-safety require in this phase. No behavioural change to validation output.
</action>

<acceptance_criteria>
- `validate.js` contains `tokensByKey` populated inside the existing content loop (one entry per `contents` key).
- `node validate.js <example-output.json>` produces byte-identical `── results ──` output to the pre-change version (before/after diff empty).
- `tokensByKey` is non-empty for the shipped example (T6 asserts `Object.keys(tokensByKey).length >= 1`, first entry is an array of token objects with `name`/`attrs`).
- No new `require` of `render-safety` is added in this phase.
</acceptance_criteria>

### T3 — `theatreAttrs()` as sole writer of the custom-attributes path (divi-builder.js)

<read_first>
- skills/landing-page/scripts/divi-builder.js  (`theatreAttrs` 96-113; `withTheatre` 116-119; `merge` 26-37; each module's `attrs = prune(merge(attrs, withTheatre(o)))`)
- skills/landing-page/RENDER-FAULT-FINDER-SPEC.md §4 "RS-ATTR-PATH" (119-123)
- memory: divi5-custom-attribute-schema (canonical path is module.decoration.attributes)
</read_first>

<action>
Add a normaliser invoked inside `withTheatre()` (or immediately after each merge) that
inspects merged attrs for a wrong-shape custom-attributes branch arriving via the
`attrs:` escape hatch and either migrates it to the canonical array or throws a clear
build-time error. Detect: (a) `module.advanced.attributes`, (b) top-level
`advanced.attributes`, (c) `module.decoration.attributes.desktop.value.attributes`
present but not an `Array` of `{name,value}` objects. Migration target identical to
`theatreAttrs()` output: `{ name, value: String(v), targetElement: 'main' }[]`. Default:
**throw** with the offending path named; migration acceptable only if byte-identical
canonical output. Correct `theatreAttrs()` output passes through untouched.
</action>

<acceptance_criteria>
- A module built with `attrs: { module: { advanced: { attributes: { 'data-theatre': 'fade-up' } } } }` throws at build time naming `module.advanced.attributes`, or emits the canonical array — never the `advanced` shape.
- A module built with `theatre: 'fade-up'` emits exactly `module.decoration.attributes.desktop.value.attributes = [{name:'data-theatre',value:'fade-up',targetElement:'main'}]` — unchanged.
- A key→value map at the decoration path is rejected/migrated, never emitted as-is.
- All existing `examples/` still assemble without error.
</acceptance_criteria>

### T4 — Confirm raw-quote mechanism, then harden content emission (divi-builder.js)

<read_first>
- skills/landing-page/scripts/divi-builder.js  (`block` 63-69; `dv` 56-60; `text` 238-258; `heading`; `blurb`; `button`; `placeholder`; `assemble` 422-448 — double-encoding for et_builder)
- skills/landing-page/RENDER-FAULT-FINDER-SPEC.md §4 "RS-RAW-QUOTE" (131-135), §2 fault #3
</read_first>

<action>
**Step 1 — confirm (gating).** Before changing emission, reproduce the fault: build a
`text({ html: '<p>data-theatre="fade-up"</p>' })`, run it through `block()` →
`assemble({context:'et_builder'})` → `JSON.stringify`, then JSON.parse back and confirm
whether the inner block-comment JSON round-trips. Record the finding (one paragraph)
under "## Findings": does fault #3 originate in the generator path at all, or only in
hand-authored / non-`block()` content? The Phase 1 RS-RAW-QUOTE auto-fix design depends
on this answer.

**Step 2 — harden.** Route all module innerContent HTML through one shared
`htmlContent(html)` normaliser used by `text`, `heading` (where HTML), `blurb`,
`button`, `eyebrow`, and the footer link path. It must make emitted HTML robust
regardless of the confirmed mechanism: convert double-quoted HTML attributes to single
quotes (matching the existing footer `<a href='…'>` convention) and escape stray literal
`"` in text nodes to `&quot;`. Idempotent; must not alter `&quot;`, `&#34;`, existing
single-quoted attributes, or `$variable(...)$` / `presetTag` (`&quot;`) tokens.
</action>

<acceptance_criteria>
- "## Findings" states the confirmed origin of fault #3 (generator vs hand-authored) with round-trip evidence.
- A `text()` with `<p>He said "hi" and href="x"</p>` emits content that, after `assemble`→`JSON.stringify`→`JSON.parse`, has no unescaped `"` inside the block-comment JSON and renders as a non-empty `<p>` (string-shape asserted in T6).
- `htmlContent()` idempotent: `htmlContent(htmlContent(x)) === htmlContent(x)` for test inputs.
- `&quot;`, single-quoted attrs, `$variable(...)$` tokens pass through unchanged.
- Existing `examples/` output still validates green.
</acceptance_criteria>
