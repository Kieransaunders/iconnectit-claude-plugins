# Step 3 — Product website / landing page

A .org listing or marketplace page is rented space with fixed structure. A
dedicated product site (or at minimum a landing page on the company site) gives
full control over content, format, and brand, and is the home for content
marketing, SEO, and email capture. Build it from the MVC assets.

## Structure: clean, flat, four pages

Keep it shallow — good for SEO and for buyers. Mirror this:

- **Home** — the pitch + primary CTA above the fold; social proof; benefit
  blocks; secondary CTA.
- **Features + Demo** — what it does, shown working. Include a short demo video
  (a buy/demo page with an explanatory video can lift conversion substantially).
- **Buy / Download** — the conversion page. If there are tiers, show them here
  clearly with the recommended tier highlighted.
- **Support / Documentation** — reuse the cornerstone pages; reassures buyers
  pre-purchase and deflects support load.

## Copy, page by page

**Home (above the fold):** tagline as H1, one-sentence sub that names the
pain-point and outcome, primary CTA button, and a trust strip (logos / install
count / rating) immediately visible. Then benefit blocks (reuse the long
description), a short "how it works" in 3 steps, testimonials, FAQ (reuse
objections asset), final CTA.

**Features + Demo:** group features by the job they do for the user, each with
the benefit first. Embed the demo. End with a CTA.

**Buy/Download:** restate the core benefit + risk-reducers (guarantee, free
version, support, GDPR/VAT handled), then the pricing table. One clear default
choice. Repeat social proof next to the buy button.

**Support/Docs:** the cornerstone pages as a clean, searchable set.

## Social proof — placement matters

Feature testimonials, "as featured on", reviews, user/download counts, and
recent-purchase popups. The most common mistake is burying testimonials just
above the footer. **Place social proof immediately next to the CTA** (the buy
button, the signup form) where it reduces hesitation at the decision point.

## Conversion + lead capture from day one

- Put a **Buy/Get-started button in the main nav** so it's visible on every page.
- Set up a simple lead-generation flow (an opt-in offering something useful — a
  guide, checklist, mini-course) and an automated welcome/checkout email
  sequence from the start, not later.
- **Segment and tag** subscribers (prospect vs trial vs customer; by interest)
  so you can send relevant automated or manual emails about: new versions & bug
  fixes, discounts, freebies, known issues, and webinars.
- GDPR: explicit opt-in, clear purpose, easy unsubscribe (UK/EU default).

## The blog — powerful, but only if resourced

A blog drives SEO traffic and educates users (which also deflects support), but
it only works with a sustained flow of posts. If the user can't commit to
regular publishing, **put the blog on hold rather than launch a stale one** —
but treat that as temporary, because without it you lose organic traffic and
fresh content. When it is on, anchor posts to the raw bits (see `channels.md`
for the content playbook) and target real search queries.

## Default email sequences to draft

When asked for "the emails", produce finished copy for:

1. **Welcome / onboarding** (free or trial signup) — get them to first value
   fast, link the Setup + demo pages.
2. **Trial-to-paid / free-to-pro** — surface the paid benefit at the moment it's
   relevant; include social proof and a clear upgrade CTA.
3. **Cart/checkout abandonment** — recover the drop-off with reassurance +
   incentive.
4. **Re-engagement** — for dormant users, tied to a new feature or use case.

Keep each email short, one job, one CTA, in the product's voice.
